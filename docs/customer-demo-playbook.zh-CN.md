# 客户演示手册

该手册用于在 15-25 分钟内稳定演示 `sas-convert-lab`，面向数据迁移相关干系人。

## 演示目标

让客户清楚看到迁移过程：

- 在 VS Code 内由开发者主导
- 由多代理协同推进
- 通过分阶段提问持续对齐业务意图

## 演示前检查清单

- 在 VS Code 打开仓库。
- 确认 Copilot Chat 可用。
- 准备一个小型 SAS 示例工程。
- 确认 `.github/agents/*.agent.md` 可见。
- 备好 `examples/sample-run/artifacts/run-001/` 作为兜底示例。

## 推荐演示脚本

## 1) 先讲框架定位（2 分钟）

可直接说：

- “这不是一次性代码翻译。”
- “这是带阶段闸门的迁移流程。”
- “全过程不依赖 SAS runtime。”

## 2) 触发总控代理（1 分钟）

输入：

```text
@sas-orchestrator Convert this SAS project to python-bigquery.
Focus on nightly ETL jobs first.
```

预期：

- orchestrator 启动 `Discover`
- 输出阶段闸门（问题 + 假设 + 对齐分）

## 3) 演示人机协同闸门（5 分钟）

现场回答 Discover/Analyze 问题：

- 明确范围内/范围外作业
- 指定数据源真值口径
- 明确 KPI 容差与 SLA 优先级

重点强调：

- 用户不确认，不进入下一阶段
- 未决问题进入 `open_questions.json`

## 4) 演示设计与转换对齐（5-8 分钟）

在进入 Convert 前展示：

- 先复述开发者愿景（pre-convert gate）
- 用户可修正假设再继续
- 若命名/边界不满意，先改再转

重点强调：

- 转换是“被引导”的，不是“盲生成”
- 源到目标映射全程可追踪

## 5) 演示验证与决策（3-5 分钟）

展示最终产物：

- `validation_report.json`
- `conversion_manifest.json`
- must-fix 与 accepted deviations 清单

收尾话术：

- “这份输出可以直接支撑 go/no-go 评审。”

## 演示中的常见追问

- “是否能先迁移高价值作业？”
- “宏逻辑复杂且置信度低时怎么处理？”
- “上线前能否把 KPI 容差进一步收紧？”

## 常见异议与回应

- **异议：** “能不能全自动一键迁移？”
  - **回应：** “可自动生成，但关键决策必须人工把关，这是风险控制设计。”
- **异议：** “需要安装 SAS 吗？”
  - **回应：** “不需要，框架只做静态与语义分析。”
- **异议：** “如何信任转换结果？”
  - **回应：** “可追踪映射 + 分阶段验证 + 风险显式输出。”

## 演示成功信号

- 干系人理解阶段闸门价值并认可流程。
- 现场开始讨论范围、容差、验收标准（说明参与到位）。
- 团队愿意以波次方式启动试点迁移。
